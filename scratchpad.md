# 基于STM32的声光报警控制系统 详细设计文档

## 1. 项目概述

本项目是一个基于STM32F10x系列微控制器的**嵌入式声光报警与状态指示系统**，模拟了“迷你温室”环境监控终端的核心功能。系统通过RGB LED指示灯、有源蜂鸣器、物理按键和串口通信，实现了多种工作状态（正常、自检警告、故障报警、巡检）的切换与提示，并具备定时自检、串口远程控制及状态上报能力。

### 主要功能
- **四色状态指示**：绿（正常）、黄（30秒周期自检警告）、红（持续故障报警）、白（手动巡检模式，持续5秒）。
- **蜂鸣器提示**：短促三声警告（WARN）和持续间断长鸣报警（ALARM）。
- **物理按键控制**：KEY1 触发/清除故障报警；KEY2 进入巡检模式（故障报警时禁用）。
- **串口远程控制**：支持 `@alarmon`、`@alarmoff`、`@inspect`、`@ledxxx` 等命令，并返回ACK/NACK响应。
- **自动周期自检**：在正常状态下每30秒触发一次黄灯+短鸣警告（持续3秒后自动恢复）。
- **状态上报**：可定时发送包含当前状态、系统运行时间的二进制数据包（本代码中默认注释，可启用）。
- **非阻塞运行**：所有延时和时序由10ms系统节拍定时器驱动，主循环无阻塞。

## 3. 软件总体架构

系统采用**前后台**架构：
- **后台（主循环）**：处理串口命令、按键事件、秒级定时任务（状态超时、周期自检、数据上报）。
- **前台（中断）**：
  - **TIM2 10ms中断**：更新系统秒计数器（`g_system_seconds`），并调用 `Buzzer_Tick_10ms()` 控制蜂鸣器时序。
  - **按键外部中断**（或EXTI）：设置 `g_key1_pressed`/`g_key2_pressed` 标志，主循环轮询处理。
  - **USART1接收中断**：将接收到的字符存入环形缓冲区，当收到 `\r\n` 或缓冲区满时置 `g_rx_ready` 标志。

模块划分清晰：`bsp_led`、`bsp_buzzer`、`bsp_key`、`bsp_timer`、`bsp_usart`、`Delay`，主程序 `main.c` 实现状态机与业务逻辑。

## 4. 系统状态机（核心）

### 4.1 状态定义
```c
typedef enum {
    STATE_NORMAL = 0,          // 正常运行（绿灯）
    STATE_SELF_CHECK_WARN,     // 30s自检预警（黄灯 + 短鸣）
    STATE_FAULT_ALARM,         // 故障持续报警（红灯 + 间断长鸣）
    STATE_INSPECTION           // 手动巡检模式（白灯常亮5s）
} SystemState_t;
```

### 4.2 状态切换逻辑
- **入口切换**：调用 `Change_State(new_state)`，该函数负责：
  - 更新当前状态变量 `g_current_state`；
  - 重置状态计时器 `g_state_timer_seconds = 0`；
  - 根据新状态设置对应的LED颜色和蜂鸣器模式；
  - 若切换至 `STATE_NORMAL`，同时重置自检基准时间 `g_last_self_check_second = g_system_seconds`。

- **切换触发条件**：
  - **定时自检**：在 `STATE_NORMAL` 下，每30秒自动切至 `STATE_SELF_CHECK_WARN`。
  - **超时恢复**：自检状态持续3秒后自动恢复至 `STATE_NORMAL`；巡检状态持续5秒后自动恢复至 `STATE_NORMAL`（二者均通过秒级计时判断）。
  - **按键触发**：KEY1 在非报警状态下触发报警，在报警状态下清除报警（切换至NORMAL）；KEY2 在非报警状态下触发巡检。
  - **串口命令**：`@alarmon` → 强制进入报警；`@alarmoff` → 恢复正常；`@inspect` → 进入巡检（若当前非报警状态）。

### 4.3 状态机示意图（文字描述）
```
        ┌─────────────────────────────┐
        │          NORMAL             │
        │  (绿灯, 蜂鸣器OFF)          │
        └──────┬──────────────┬───────┘
               │              │
    (每30秒)   │              │ (KEY1/串口alarmon)
               ▼              ▼
    ┌──────────────────┐  ┌────────────┐
    │ SELF_CHECK_WARN │  │FAULT_ALARM │
    │ (黄灯, WARN鸣叫)│  │(红灯, ALARM)│
    └──────┬───────────┘  └─────┬──────┘
           │ (3秒后)            │ (KEY1/串口alarmoff)
           └──────────┬─────────┘
                      ▼
                (返回NORMAL)
           (INSPECTION 从NORMAL或SELF_CHECK进入，5秒后返回)
```

## 5. 模块化驱动详解

### 5.1 LED驱动 (`bsp_led.c`)
- **初始化**：使能GPIO时钟，配置三个引脚为推挽输出，默认关闭（`LED_Off()` 将所有引脚置高，因为共阳极低电平点亮）。
- **颜色控制函数**：
  - `LED_Green()`：绿=低，红/蓝=高
  - `LED_Yellow()`：红+绿=低，蓝=高
  - `LED_Red()`：红=低，绿/蓝=高
  - `LED_White()`：红绿蓝全低
  - `LED_Off()`：全高
> 颜色组合依赖硬件共阳极，代码中通过位操作实现。

### 5.2 按键驱动 (`bsp_key.c`)
- **初始化**：配置两个GPIO为输入（浮空或上拉），并使能EXTI中断（下降沿触发）。
- **中断服务**：在EXTI中断中消抖（通常通过软件延时或状态机），置位全局标志 `g_key1_pressed` / `g_key2_pressed`，并清除中断挂起位。
- **主循环处理**：轮询标志，执行相应动作后清零。

### 5.3 蜂鸣器驱动 (`bsp_buzzer.c`)
- **初始化**：GPIO推挽输出，初始低电平（关闭）。
- **模式控制**：`Buzzer_SetMode(mode)` 切换模式，重置计数器。
- **中断节拍函数**：`Buzzer_Tick_10ms()` 由10ms定时器中断调用，实现无阻塞时序：
  - **WARN模式**：150ms响 → 150ms停，重复3次后自动关闭。
  - **ALARM模式**：500ms响 → 500ms停，无限循环。
- **设计要点**：所有状态变量均声明为 `volatile`，确保中断与主循环数据可见性。

### 5.4 定时器驱动 (`bsp_timer.c`)
- **TIM2配置**：预分频器7200-1（得到10kHz计数时钟），自动重载值100-1（产生100个计数周期，即10ms）。
- **中断服务**：每10ms进入一次，执行：
  1. `g_system_seconds` 每满100次（即1秒）递增一次（在中断中实现秒计数）。
  2. 调用 `Buzzer_Tick_10ms()`。
- 秒计数变量 `g_system_seconds` 为 `volatile uint32_t`，供主循环使用。

### 5.5 串口驱动 (`bsp_usart.c`)
- **初始化**：设置波特率115200，8数据位，1停止位，无校验；使能接收中断和发送功能。
- **接收中断**：每次接收一个字符存入 `g_rx_buf`，当检测到 `\r\n` 或缓冲区满（例如64字节）时置 `g_rx_ready=1`，并记录长度。
- **发送函数**：`USART1_SendString()` 阻塞轮询发送（适合调试），`USART1_SendDataPacket()` 发送二进制数据。
- 采用全局缓冲区，支持命令解析（以 `@` 开头的字符串命令）。

### 5.6 延时驱动 (`Delay.c`)
- 基于SysTick或简单循环实现微秒/毫秒级延时，仅在初始化阶段短暂使用（如上电LED闪烁），运行后不再使用阻塞延时。

## 6. 中断与定时机制

- **10ms节拍**：所有周期性动作（蜂鸣器时序、秒计数）均依赖TIM2中断，保证了实时性和非阻塞性。
- **主循环轮询**：按键标志、串口命令标志在 `while(1)` 中处理，无阻塞延时。
- **秒级任务**：通过比较 `g_system_seconds` 与上次值，在主循环中执行每秒一次的任务（状态超时判断、周期自检触发、数据上报）。
- **中断优先级**：应确保TIM2中断优先级高于串口和按键，避免蜂鸣器时序抖动（具体在初始化NVIC中配置）。

## 8. 主循环逻辑（每秒任务）

```c
while (1) {
    Process_Serial_Command();   // 处理串口命令
    if (g_key1_pressed) { ... } // 处理按键事件
    if (g_key2_pressed) { ... }

    // 每秒执行一次的任务
    if (g_system_seconds != last_seconds) {
        last_seconds = g_system_seconds;

        // 1) 有限状态超时处理（INSPECTION 5秒，SELF_CHECK 3秒）
        if (g_current_state == STATE_INSPECTION) {
            g_state_timer_seconds++;
            if (g_state_timer_seconds >= 5) Change_State(STATE_NORMAL);
        }
        else if (g_current_state == STATE_SELF_CHECK_WARN) {
            g_state_timer_seconds++;
            if (g_state_timer_seconds >= 3) Change_State(STATE_NORMAL);
        }

        // 2) 周期自检触发（仅在NORMAL状态，每30秒）
        if (g_current_state == STATE_NORMAL) {
            if (g_system_seconds - g_last_self_check_second >= 30) {
                Change_State(STATE_SELF_CHECK_WARN);
            }
        }

        // 3) 可选：每10秒上报一次二进制包
        // if (g_system_seconds % 10 == 0) Send_Binary_Packet();
    }
}
```

## 9. 关键函数剖析

### `Change_State(SystemState_t new_state)`
- 核心状态切换函数，原子性更新状态、重置计时器、调用LED/蜂鸣器设置。
- 特别处理：切至NORMAL时重置自检基准，避免刚恢复又立即触发自检。

### `Process_Serial_Command()`
- 检查 `g_rx_ready`，解析接收缓冲区中的字符串。
- 使用 `strcmp` 匹配预定义命令，执行相应操作，并返回ACK/NACK。
- 清空接收标志，准备下一帧。

### `Buzzer_Tick_10ms()`
- 在中断上下文中运行，必须简短。
- 根据 `g_buzzer_mode` 和内部计数器产生精确的响/停时序，切换引脚电平。
- WARN模式自动结束并关闭蜂鸣器，ALARM模式无限循环。

## 10. 系统启动流程

1. 调用各外设初始化函数（LED、Buzzer、Key、USART）。
2. 上电自检：白灯亮500ms，串口打印启动信息。
3. 设置初始状态为 `STATE_NORMAL`（绿灯亮，蜂鸣器关，重置自检基准）。
4. 初始化TIM2并开启中断（10ms节拍）。
5. 进入主循环，处理事件。

## 11. 答辩常见问题与解答

### Q1: 为什么采用状态机设计？
**A**: 状态机将系统行为划分为有限个离散状态，每个状态有明确的LED/蜂鸣器响应和转移条件，使逻辑清晰、易于维护和扩展。同时，状态机与事件驱动（定时、按键、串口）结合，能高效管理复杂时序。

### Q2: 蜂鸣器驱动为什么放在中断里？如何保证不阻塞？
**A**: 蜂鸣器需要精确的时序（如150ms响/停），若在主循环中用延时则会阻塞其他任务。将时序控制放在10ms中断中，仅改变计数器并置位/复位引脚，主循环无需等待，实现了完全非阻塞。

### Q3: 如何避免按键抖动？
**A**: 代码中未显示具体消抖，但通常可以在EXTI中断中采用软件延时（如10ms）或使用状态机检测稳定电平。此处可通过按键中断触发后，在主循环中延时读取电平确认，或者直接利用10ms节拍进行定时扫描（建议改进）。

### Q4: 系统时间 `g_system_seconds` 在中断中累加，主循环读取时会不会出现竞争？
**A**: `g_system_seconds` 是32位变量，在中断中每10ms判断是否溢出（满100次）才加1，实际上只在每秒加1，而主循环每秒检查一次。即使发生读-修改-写冲突（因为中断可能打断主循环的读操作），但读取32位变量在STM32上是原子操作（对齐访问），且加1操作在中断中完成，主循环只读，因此安全。但更严谨的做法可加入临界区。

### Q5: 串口命令解析使用 `strcmp` 是否安全？
**A**: `strcmp` 需要确保字符串以 `'\0'` 结尾。接收中断中，我们以 `\r\n` 作为结束符，并在缓冲区末尾置 `'\0'`（代码未显示但假设实现），因此安全。同时限制缓冲区大小防止溢出。

### Q6: 若系统处于故障报警状态，为何禁止进入巡检模式？
**A**: 业务逻辑上，故障报警是最高优先级，需要持续声光警示，而巡检模式会改变LED和关闭蜂鸣器，可能掩盖报警，因此设计为报警状态下巡检命令被拒绝（返回NACK）。

### Q7: 如何扩展新的状态或命令？
**A**: 
- 新状态：在枚举中添加，在 `Change_State` 的 `switch` 中添加对应的LED和蜂鸣器设置，并在主循环的秒任务中添加相应的超时或转移逻辑。
- 新命令：在 `Process_Serial_Command` 中添加 `else if` 分支，匹配字符串并调用相应函数。

### Q8: 为什么上电自检只闪白灯500ms，而没有蜂鸣器测试？
**A**: 设计简约，可通过添加短暂蜂鸣器鸣叫来增强自检，只需在初始化后调用 `Buzzer_SetMode(BUZZER_MODE_WARN)` 并延时，但需注意不要阻塞启动流程。

### Q9: 定时器中断周期为什么选10ms？有什么考虑？
**A**: 10ms是常见的嵌入式系统节拍，能够满足蜂鸣器最小时间精度（150ms=15个节拍）和人眼可察觉的LED变化，同时中断开销适中。过短（如1ms）会增加CPU负担，过长（如50ms）则无法精细控制蜂鸣器时序。

### Q10: 代码中 `Send_Binary_Packet()` 被注释了，如何启用？
**A**: 在主循环每秒任务中取消注释 `if (g_system_seconds % 10 == 0) Send_Binary_Packet();`，并确保 `USART1_SendDataPacket` 已实现，即可每10秒自动上报状态。
