### 第一部分：内核核心机制 (Kernel Depth)

#### 1. 线程管理 (Thread Management)
线程是 RT-Thread 中最基本的调度单位。
*   **线程组成**：每个线程由 **线程控制块 (rt_thread_t)**、**栈 (Stack)** 和 **入口函数** 组成。
*   **优先级 (Priority)**：RT-Thread 支持 256 个优先级（通常用 32 个）。**0 级最高**，数值越大优先级越低。
*   **时间片 (Tick)**：仅对**相同优先级**的线程有效。当多个同级线程就绪时，按时间片轮转。
*   **状态切换**：
    *   *初始 (Init)* -> *就绪 (Ready)* -> *运行 (Running)* -> *挂起 (Suspend)* -> *关闭 (Close)*。
*   **静态 vs 动态**：
    *   **静态线程**：`rt_thread_init`。栈空间是编译时分配好的数组。优点：内存分配确定，不会失败。
    *   **动态线程**：`rt_thread_create`。从内存堆中申请空间。优点：灵活，线程结束后自动释放空间。
*   **空闲线程 (Idle Thread)**：系统最低优先级的线程，永远处于就绪态。当没有其他线程运行时，它负责回收已删除线程的资源，或让 CPU 进入省电模式。

#### 2. 线程同步 (Synchronization)
解决线程间“竞争”或“时序配合”的问题。
*   **信号量 (Semaphore)**：
    *   **原理**：一个计数器。`take` 使计数器减 1，`release` 使计数器加 1。
    *   **应用**：主要用于**同步**。例如：中断告知线程“数据已到达”。
*   **互斥量 (Mutex)**：
    *   **核心特性**：**所有权**（谁加锁谁解锁）和**优先级继承机制**。
    *   **优先级继承**：防止“优先级翻转”。如果高优先级线程等待低优先级线程持有的锁，系统会临时提升低优先级线程的等级。
    *   **应用**：**资源保护**。例如：防止两个线程同时打印串口导致字符乱掉。
*   **事件集 (Event)**：
    *   **特点**：一个 32 位的标志位，支持“逻辑与（所有条件满足）”和“逻辑或（任一条件满足）”。

#### 3. 线程间通信 (IPC)
解决线程间“传递数据”的问题。
*   **邮箱 (Mailbox)**：
    *   **限制**：每封邮件固定 **4 字节**。
    *   **高效性**：非常适合传递一个**指针**或一个简单的整数状态。
*   **消息队列 (Message Queue)**：
    *   **特点**：可以传递不定长度的数据块，内部有缓冲区。
    *   **应用**：生产者-消费者模型，如传感器连续采集数据并发送给处理线程。

### 第二部分：内存与中断 (System Safety)

#### 1. 内存管理 (Memory Management)
RT-Thread 将内存分为**静态内存池**和**动态内存堆**。
*   **动态堆 (Heap)**：`rt_malloc` 和 `rt_free`。
    *   **碎片化问题**：频繁申请不同大小的内存会导致堆中出现很多无法利用的小空隙。
    *   **面试考点**：为什么不能在中断中 malloc？因为 malloc 涉及链表操作且可能有互斥锁，而中断不能被阻塞，且 malloc 执行时间不确定。
*   **内存池 (Pool)**：申请固定大小的块。速度极快，无碎片，常用于高频数据传输。

#### 2. 中断管理 (Interrupt Management)
中断是系统的最高优先级，可以打断任何线程。
*   **中断上下文 (ISR)**：
    *   **限制**：**严禁使用任何会导致线程挂起的函数**（如 `rt_thread_mdelay` 或 `rt_sem_take` 等待模式）。
    *   **原则**：快进快出，只做最简单的数据拷贝或标志位翻转。
*   **栈使用**：在 ARM Cortex-M 中，线程使用 **PSP**（进程栈指针），中断使用 **MSP**（主栈指针）。这种分离保证了即使某个线程栈溢出，中断依然能正常运行。

### 第三部分：标准设备驱动框架 (RT-Device)

这是 RT-Thread 的精髓，它将硬件操作抽象成了类似 Linux 的“文件操作”。

#### 1. 框架层次
1.  **应用层**：调用标准 API (`rt_device_read` 等)。
2.  **I/O 设备管理层**：负责按名称查找设备。
3.  **设备驱动层**：对接具体芯片外设（如 STM32 HAL 库）。

#### 2. 标准操作流程
```c
rt_device_t dev = rt_device_find("uart1"); // 1. 找设备
rt_device_open(dev, RT_DEVICE_FLAG_INT_RX); // 2. 打开设备（开启中断接收）
rt_device_set_rx_indicate(dev, rx_callback); // 3. 设置接收回调
rt_device_read(dev, 0, buf, size);          // 4. 读取数据
rt_device_close(dev);                       // 5. 关闭设备
```

#### 3. 常见设备细节
*   **PIN 设备**：最简单的 GPIO 操作，支持中断绑定。
*   **UART 设备**：工业常用，配合 `rt_completion` 或 `rt_sem` 实现异步高效接收。
*   **I2C/SPI 总线**：RT-Thread 将总线和设备分开。你需要先通过 `rt_hw_i2c_init` 注册总线，然后应用层通过 `rt_i2c_transfer` 进行通信。

### 第四部分：高级组件与工具 (Ecosystem)

#### 1. FinSH 命令行
这是 RT-Thread 最好用的调试工具。
*   **常用命令**：
    *   `list_thread`：查看线程状态、优先级、栈使用率（防止爆栈）。
    *   `list_sem` / `list_mutex`：查看同步对象。
    *   `free`：查看内存堆剩余空间。
*   **自定义命令**：使用 `MSH_CMD_EXPORT(name, desc)` 即可把自己的函数变成命令行命令。

#### 2. 构建系统：SCons & Kconfig
*   **Kconfig**：负责“配置”。在 RT-Thread Studio 的图形化界面勾选某个功能，背后就是在修改 `rtconfig.h`。
*   **SCons**：负责“编译”。基于 Python 的脚本，自动扫描项目中的 `.c` 文件并生成 Makefile。

#### 3. DFS & SAL
*   **DFS (Virtual File System)**：让你像在电脑上一样操作 `sdcard/data.txt`。
*   **SAL (Socket Abstraction Layer)**：抽象了网络套接字。无论底层是 WiFi 模块还是以太网卡，应用层代码都长得一样。

### 第五部分：面试高频逻辑与实战复盘

#### **1. 阻塞与死锁 (Blocking & Deadlock)**
*   **现象**：程序卡死，不再打印或响应。
*   **预防**：
    *   中断内绝不阻塞。
    *   加锁顺序一致（线程 A 拿锁 1 再拿锁 2，线程 B 也必须是 1 之后 2，严禁反过来）。
    *   互斥量设置超时时间，不要 `RT_WAITING_FOREVER`。

#### **2. 优先级翻转 (Priority Inversion)**
*   **场景**：高优先级线程 H 等待低优先级线程 L 释放锁，此时中优先级线程 M 抢占了 L 的 CPU，导致 H 被迫等待 M，造成系统逻辑混乱。
*   **解决**：使用 **互斥量 (Mutex)**。RT-Thread 会临时把 L 的优先级提升到和 H 一样，直到 L 释放锁。

#### **3. 为什么选择 RTOS 而不是裸机？**
*   **裸机 (Bare Metal)**：大循环 + 中断。当功能复杂时，逻辑耦合严重，难以维护，实时性差。
*   **RTOS**：多线程并发。将任务解耦，高优先级任务可以随时抢占，大大提高了复杂系统的稳定性和实时性。

#### **4. 消息队列 vs 全局变量**
*   **全局变量**：没有同步机制。数据写了一半可能被读走（乱码），且无法通知接收方“数据来了”。
*   **消息队列**：内置同步和缓冲。生产者发送后，消费者立刻从挂起态唤醒，且数据按顺序排列，不会丢失。
