在 FreeRTOS 中，**事件组（Event Groups）** 是一种非常强大且灵活的线程间同步机制。如果你之前学习过 RT-Thread，你会发现 FreeRTOS 的事件组与 RT-Thread 的“事件集（Event Set）”在逻辑上几乎是一模一样的。

简单来说，事件组就是一组 **“标志位（Bit Flags）”**，每一个位代表一个事件是否发生。

---

### 1. 事件组的核心概念

*   **位（Bits）的概念**：
    通常一个事件组由一个 `EventBits_t` 类型的变量表示。
    *   在大多数 32 位处理器上，FreeRTOS 的事件组包含 **24 个可用的位**（另外 8 位被内核用来存储控制信息）。
    *   每一位代表一个“事件状态”。例如：
        *   Bit 0：传感器 A 数据采集完成。
        *   Bit 1：按键被按下。
        *   Bit 2：网络连接成功。

*   **多对多同步**：
    *   **信号量**通常只能一个任务等另一个任务。
    *   **事件组**可以让一个任务等待**多个事件同时满足**（AND 逻辑），或者**其中任一满足**（OR 逻辑）。
    *   **广播效应**：当一个事件发生并被触发时，所有正在等待该事件的任务都会被唤醒（不像信号量，一次只能唤醒一个任务）。

---

### 2. 事件组的关键 API

#### (1) 创建事件组
```c
EventGroupHandle_t xEventGroup;
xEventGroup = xEventGroupCreate();
```

#### (2) 设置事件位（发送事件）
*   **在线程中调用**：
    ```c
    // 设置 Bit 0 和 Bit 1
    xEventGroupSetBits(xEventGroup, (1 << 0) | (1 << 1));
    ```
*   **在中断中调用**（FreeRTOS 严格区分中断 API）：
    ```c
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xEventGroupSetBitsFromISR(xEventGroup, (1 << 0), &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    ```

#### (3) 等待事件位
这是最常用的 API，参数比较多：
```c
EventBits_t uxBits;
uxBits = xEventGroupWaitBits(
            xEventGroup,        // 事件组句柄
            (1 << 0) | (1 << 1),// 等待哪些位
            pdTRUE,             // 退出时是否自动清除位（设为 pdTRUE 则唤醒后位清零）
            pdTRUE,             // 是否等待所有位都满足（pdTRUE 为 AND，pdFALSE 为 OR）
            portMAX_DELAY       // 等待时间
         );
```

### 3. 事件组的常见应用场景

#### 场景 A：逻辑“与（AND）”同步（多对一）
比如一个复杂的初始化流程：
*   任务 1：初始化 SD 卡。
*   任务 2：初始化 WiFi。
*   任务 3：初始化 UI。
*   **主控任务**：必须等以上 3 个任务全部初始化完成（Bit 0 & Bit 1 & Bit 2 都置位），才能进入下一步。

#### 场景 B：逻辑“或（OR）”同步
比如一个报警系统：
*   位 0：烟雾报警。
*   位 1：入侵报警。
*   位 2：温度过高报警。
*   **报警线程**：只要其中**任何一个**位置位（OR 逻辑），就立即报警。

#### 场景 C：任务同步点（Rendezvous）
FreeRTOS 有一个特殊的函数 `xEventGroupSync()`。
*   **功能**：让多个任务在某个点“集合”。
*   **逻辑**：任务 A 到达同步点，设置自己的位并等待其他人的位；直到所有任务都到达，所有任务才同时继续执行。这在并行计算中非常有用。

---

### 4. 重点：事件组 vs 信号量

| 特性 | 二值信号量 (Binary Semaphore) | 事件组 (Event Groups) |
| :--- | :--- | :--- |
| **存储数据** | 无（只有 0 和 1） | 24 个独立的标志位 |
| **等待逻辑** | 只能等待一个信号 | 可以等待 AND 或 OR (多条件) |
| **唤醒机制** | 只能唤醒一个等待的任务 | **广播**：唤醒所有等待该条件的任务 |
| **资源消耗** | 极低 | 稍高（因为要维护链表扫描哪些任务条件满足） |
| **中断支持** | 较快 | `FromISR` 函数实际上是通过定时器任务间接处理的，响应稍慢 |

---

### 5. 注意事项（避坑指南）

1.  **中断中的延迟处理**：
    在 FreeRTOS 中，`xEventGroupSetBitsFromISR` 并不是在中断里直接修改位，而是给一个“定时器任务”发消息，让它在中断结束后去改。因此，事件组在中断里的实时性略逊于信号量。
2.  **不具备计数功能**：
    事件组是“幂等”的。如果你连续设置两次 Bit 0，而没有任务读取，它仍然只是 1 个 Bit 0。如果你需要记录事件发生的次数，请使用 **计数信号量 (Counting Semaphore)**。
3.  **自动清零**：
    在 `xEventGroupWaitBits` 中，如果设置了 `pdTRUE`（退出清除），那么只有当满足等待条件并成功返回时，位才会被清除。如果不满足条件超时退出，位不会被清除。

### 总结
**FreeRTOS 事件组** 就是一种 **“多信号复合管理工具”**。它最适合处理那种“当 A 发生且 B 发生或 C 发生时，执行任务 D”的复杂逻辑。如果你在项目中发现自己用了好几个信号量来控制一个线程，那就该考虑换成事件组了。
